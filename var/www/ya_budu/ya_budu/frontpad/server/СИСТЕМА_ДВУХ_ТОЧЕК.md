# Исправления системы двух точек

## Проблемы, которые были исправлены:

### 1. **WebSocket рассылка заказов**
**Проблема:** Все админы получали ВСЕ заказы, независимо от point_id.

**Решение:** Исправлена логика broadcast:
```javascript
// ПРАВИЛЬНАЯ логика:
if (client.pointId === 0) { // Главный админ - получает ВСЕ
  client.send(message);
  return;
}

if (client.role === 'admin' && client.pointId && data.order.point_id === client.pointId) {
  // Админ точки получает ТОЛЬКО заказы своей точки
  client.send(message);
  return;
}

if (client.role === 'courier' && client.pointId && data.order.point_id === client.pointId) {
  // Курьер получает ТОЛЬКО заказы своей точки
  client.send(message);
  return;
}
```

### 2. **API фильтрация заказов**
**Проблема:** Заказы могли отображаться у всех точек после изменения статуса.

**Решение:** Исправлено получение обновленного заказа перед broadcast:
```javascript
// Получаем обновленный заказ из базы данных
const updatedOrder = await get('SELECT * FROM orders WHERE id = ?', [id]);
updatedOrder.items = parseOrderItems(updatedOrder.items);

broadcast({ type: 'order_status_changed', order: updatedOrder });
```

## Как работает система теперь:

### Авторизация:
- **admin / admin** → point_id = 1 → видит ТОЛЬКО заказы точки 1
- **admin2 / admin2** → point_id = 2 → видит ТОЛЬКО заказы точки 2

### Распределение заказов:
1. Пользователь выбирает район доставки на основном сайте
2. Система автоматически определяет `point_id` района
3. Заказ синхронизируется с Frontpad с правильным `point_id`
4. Заказ попадает **ТОЛЬКО** в нужную точку
5. Сотрудники других точек этот заказ не видят

### WebSocket уведомления:
- Новый заказ приходит **ТОЛЬКО** сотрудникам нужной точки
- Обновления статуса приходят **ТОЛЬКО** сотрудникам нужной точки
- Админы не видят заказы друг друга

### API фильтры:
- `/api/orders` автоматически фильтрует по point_id пользователя
- `/api/delivery-zones` показывает только районы своей точки
- Все отчеты фильтруются по точке
