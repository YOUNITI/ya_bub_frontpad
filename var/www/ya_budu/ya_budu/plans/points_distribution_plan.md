# План разделения заказов между точками (Frontpad)

## Текущая ситуация

Система сейчас:
- **Единый Frontpad** - админ-панель для всех заказов
- **Несколько сайтов** - могут быть разные домены, заказы синхронизируются в Frontpad
- **Все заказы в одной базе** - нет разделения по точкам
- **Зоны доставки** - уже существуют (`delivery_zones`)

## Проблема

При открытии второй точки все заказы будут видны всем - возникнет путаница.

## Предлагаемое решение

### 1. Архитектура

```
Сайт (домен) → Frontpad (точка)
     ↓
Зона доставки → point_id → Заказ привязан к точке
```

### 2. База данных

Добавляем поле `point_id` в таблицы:

- `delivery_zones` - каждая зона принадлежит точке
- `orders` - заказ автоматически получает point_id от зоны доставки
- `products` - опционально, если меню будет разным
- `couriers` - курьеры привязаны к точке
- `points` - новая таблица для хранения точек

### 3. Логика работы

#### При создании заказа с сайта:
1. Клиент выбирает зону доставки
2. Зона доставки привязана к конкретной точке (`point_id`)
3. При создании заказа автоматически назначается `point_id`
4. Заказ попадает только в нужную точку

#### При создании заказа в Frontpad (админ):
1. Админ выбирает зону доставки или точку
2. Заказ создаётся с привязкой к точке

#### В интерфейсе Frontpad:
1. Добавить переключатель/фильтр "Точка"
2. Показывать только заказы выбранной точки
3. Возможность видеть все точки (для админа)

## Примеры API

### Получить точки
```json
GET /api/points
Response: [{"id": 1, "name": "Точка 1"}, {"id": 2, "name": "Точка 2"}]
```

### Получить заказы с фильтром по точке
```json
GET /api/orders?point_id=1
```

### Получить зоны доставки с привязкой к точке
```json
GET /api/delivery-zones
Response: [{"id": 1, "name": "Центр", "point_id": 1, ...}, {"id": 2, "name": "Район А", "point_id": 2, ...}]
```
